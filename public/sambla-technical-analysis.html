1.2 RAG in the Overall System

RAG is the central intelligence layer. Every channel (voice, chat, widget, WhatsApp, Facebook) queries KnowledgeSearchService before generating a response. It sits between intent detection and prompt generation, injecting retrieved context into system prompts.

1.4 Orchestration Logic

IntentOrchestratorService detects all intents simultaneously (not first-match) with confidence scores:

  • order_lookup: 0.9 — highest priority
  • category_recommendation: 0.85
  • product_search: 0.7
  • knowledge_query: 0.5 — almost always fires

Post-pipeline: lead scoring → handoff detection → context merging. Fallback: if orchestrator fails → legacy sequential pipeline.

2.1 Ingestion

Entry Points

  • Manual upload via KnowledgeController::store() — PDF, DOCX, TXT, CSV, URL, raw text
  • Batch sync via ProcessKnowledgeBatch — WooCommerce connector, crawlers
  • CLI via knowledge:process and knowledge:reindex commands

Chunking Strategy

Algorithm: Split by double-newlines (semantic boundaries) → accumulate paragraphs until token budget → overlap via backward word walk. Token estimation: strlen($word) / 4.

Deduplication

Key = (bot_id, title, source_type). Re-uploading same document replaces ALL previous chunks atomically via lockForUpdate() → delete → insert in a transaction.

Metadata

JSON column for arbitrary metadata. Structured fields: category (filterable), source_date (filterable), source_type, source_id.

2.4 Retrieval — Step-by-Step

1. expandQuery(query)
   +-- Dictionary synonym lookup (13 Romanian synonym groups)
   +-- Diacritic normalization (a->a, s->s, t->t)
   +-- Optional LLM rewrite (gpt-4o-mini, disabled by default)

2. generateEmbedding(query)
   +-- Cache check: "query_embedding_{botId}_{version}_{md5}" (24h TTL)
   +-- Hit: use cached | Miss: OpenAI API call, cache result

3. Hybrid SQL (single query, 3 CTEs):
   +-- CTE vector_search:
   |     1 - (embedding <=> :embed) AS similarity
   |     WHERE bot_id = :id AND status = 'ready'
   |     ORDER BY cosine distance LIMIT 20
   |
   +-- CTE fts_search:
   |     ts_rank_cd(tsvector, tsquery) * 3.0 (title) + 1.0 (content)
   |     WHERE @@ to_tsquery('romanian', 'word1 | word2 | word3')
   |     LIMIT 20
   |
   +-- FULL OUTER JOIN + RRF:
         rrf_score = 1.0/(60 + rank_v) + 1.5/(60 + rank_f)
         WHERE similarity >= 0.68 OR fts match exists

4. Post-processing:
   +-- Quality filter: similarity >= 0.68 OR rrf_score > 1/61
   +-- Dedup: max 3 chunks per document title
   +-- Optional LLM reranking (gpt-4o-mini, 20 candidates)
   +-- Return top N results

5. buildContext():
   +-- Near-duplicate removal (>80% word overlap)
   +-- Group by document title, sort by chunk_index
   +-- Format: "--- [title] [source] (relevance: X%) ---"
   +-- Stop at max_context_chars (6000 default)

Fallback chain: Embedding fails → FTS-only search → FTS fails → empty array (never crashes).

2.5 Context Building

buildContext() assembles the final RAG context string:

  1. Call search() to get ranked results
  2. Remove near-duplicates (>80% word overlap, O(n²) comparison)
  3. Group chunks by source document title, maintain relevance order
  4. Sort chunks within same document by chunk_index (reading order)
  5. Format each with header: "--- [title] [source_type] (relevance: X%) ---"
  6. Stop accumulating when max_context_chars budget reached

2.6 Generation

Prompt Structure (in order)

  1. Bot's system_prompt (tenant-configured)
  2. Product rules (if WooCommerce enabled)
  3. Knowledge context (RAG output)
  4. Extra context (orders, products, lead prompts)
  5. Policy instructions (tone, verbosity, CTA)
  6. Guardrails (ALWAYS LAST — highest LLM priority)

Anti-Hallucination Guardrails

Rules (Romanian):
  • Respond STRICTLY based on provided context
  • If information not in context, say clearly: "Nu am aceasta informatie disponibila"
  • NEVER invent prices, specs, delivery terms, dates or numbers
  • If unsure, offer to redirect to human operator
Voice-specific: Max 1-2 sentences. Spell out prices in words. Ask for clarification on transcription errors.

Model Selection (ChatModelRouter)

  • Cost budget: 15¢/conversation → fast tier when exceeded
  • Circuit breaker: >80% failure rate over 5min → swap provider
  • Voice bias: fast tier unless complex query
  • Short continuations (<8 words): fast tier
  • Provider fallback: OpenAI ↔ Anthropic bidirectional

3. Multi-Tenant Architecture

Isolation Implementation

  • Model layer: BelongsToTenant trait + TenantScope global scope on 25+ models. Automatically adds WHERE tenant_id = ?.
  • Service layer: KnowledgeSearchService filters by bot_id in raw SQL (parameterized). ProductSearchService similarly filters by bot_id.
  • Vector search: HNSW index is global, but SQL WHERE clause bot_id = :id filters BEFORE vector ranking. No cross-tenant embedding leakage.
  • Cascade behavior: All foreign keys use cascadeOnDelete().

Identified Risks

4. Strengths

1. Hybrid Search with RRF — Genuinely Well-Implemented

Not just "we do vector + keyword." The RRF formula with tunable fts_weight=1.5, Romanian-specific to_tsvector, title weighting at 3x, and HNSW indexing is a production-grade implementation. Most SaaS RAG systems skip FTS entirely.

2. 4-Layer Database Protection System

bootstrap.php blocks test execution in production → TestCase::setUp() validates DB name → AppServiceProvider prohibits destructive commands → SafeguardDatabase listener intercepts dangerous artisan commands. Paranoid-level protection most startups don't have.

3. Source-Type-Aware Chunking

FAQ=128, manual=256, upload=512 tokens. Uncommon and valuable — most systems use fixed chunk sizes regardless of content type. FAQ documents benefit from small chunks; manuals need larger semantic windows.

4. Voice-Specific RAG Optimization

3 results / 3000 chars for voice vs 5-8 / 6000 for text. Mid-call dynamic session.update with context deduplication. Product search BEFORE knowledge search (DB query < embedding API latency). Shows real voice UX thinking.

5. Multi-Stage Quality Filtering

Vector threshold (0.68) → RRF fusion → dedup (3/doc) → near-duplicate removal (80%) → optional LLM reranking. Each stage independently tunable. Cascading fallback ensures availability.

6. Intent-Aware RAG Skipping

shouldSkipKnowledge() detects greetings/thanks/followups and skips the RAG pipeline. Saves ~200-500ms latency + embedding costs on trivial messages.

7. Commerce Attribution Engine

3-mode attribution (strict/probable/assisted), idempotent event tracking, opportunity scoring, and conversation outcome derivation. E-commerce analytics done right.

8. Atomic Chunk Replacement

lockForUpdate() → delete old → insert new in a transaction. Prevents duplicate/orphan chunks during re-uploads.

5. Weaknesses / Gaps

1. Token Estimation is Crude — strlen / 4

File: ProcessKnowledgeDocument.php:411-418

For Romanian (diacritics, longer words), this underestimates by 15-25%. Chunks overshoot token limits, consuming more prompt budget than planned. tiktoken or a proper tokenizer should be used.

2. No Chunk Quality Validation

After chunking, zero validation that chunks are semantically meaningful. A chunk could be pure whitespace, a table row fragment, or a header-only fragment. No minimum content length, no coherence check.

3. ToolRegistry NOT Wired Into ChatCompletionService

ToolRegistry defines function-calling schemas but ChatCompletionService::complete() never passes tools or tool_choice parameters. The LLM can't dynamically call tools — everything is pre-fetched and stuffed into the prompt. Fundamental architectural gap.

4. No Embedding Drift Detection

When the embedding model is updated/replaced, all stored embeddings become incompatible. No versioning per chunk, no migration strategy, no cosine similarity sanity checks.

5. FTS Query Construction is Fragile

Romanian stopwords not explicitly removed. No phrase matching (<-> operator) — "pret livrare" and "livrare pret" are equivalent. Compound words not handled.

6. Hallucination Risk in Product Context

Product data injected includes price and sale_price, but no stock status, no availability date, no variant info. The LLM can confidently state a product is available when it's out of stock.

7. No RAG Evaluation Pipeline in CI

SearchEvaluate command exists for manual testing, but no automated quality regression. No golden dataset, no minimum recall/precision thresholds in CI.

8. No Conversation Summarization

20 messages consume 2000-4000 tokens of history. No summarization of earlier turns. The LLM loses context and RAG budget is wasted.

9. Single Embedding Model, Reranking Disabled by Default

text-embedding-3-small is fast and cheap but lower quality. For production SaaS where answer quality = revenue, this is underinvesting in retrieval quality.

10. Category/Source_Date Filters Are Dead Code

The $filters parameter exists in search(), but no caller passes filters. Metadata columns exist but are unused in practice.

9. Voicebot-Specific Evaluation

Voice Latency Breakdown

STT (Twilio/OpenAI)                   ~500ms
    |
Product Search (trigram DB)            ~30ms
    |
Knowledge Search (embed + pgvector)    ~200-500ms
    |
Context Build                          ~5ms
    |
session.update (WebSocket)             ~50ms
    |
LLM Generation (OpenAI Realtime)       ~500-1500ms
    |
TTS (streaming, first byte)            ~200ms
---------------------------------------------
TOTAL                                  ~1.5-2.8s

Acceptable for conversational voice (humans tolerate ~2s pauses), but marginal. Any added latency (LLM reranking, query rewrite) would break the experience.

What's Optimized

  • Lean initial context: 3 results / 3000 chars (vs 5-8 / 6000 for text)
  • Product search BEFORE knowledge search (DB trigram < embedding API latency)
  • Mid-call session.update with context deduplication (skip if unchanged)
  • search_products function tool for OpenAI Realtime dynamic search
  • Voice guardrails: max 1-2 sentences, spell out prices in words

What's NOT Optimized

  • No streaming RAG — context fetched entirely before response starts
  • No pre-warming of common queries at session start
  • No query intent classification for voice transcription errors
  • Knowledge search uses same embedding model as text (no voice tuning)

Critical Risk: Embedding API as Bottleneck

If OpenAI embedding API latency spikes (500ms → 2000ms), voice response stalls. FTS fallback exists but degrades quality. Pre-computing embeddings for common query patterns per bot would mitigate this.
