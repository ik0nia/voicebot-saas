1.3 Sesiune (Session Management)

Reluare sesiune existentă

Dacă clientul trimite session_id + session_token:

  1. Calculează expected_token = HMAC-SHA256(session_id + channelId, APP_KEY)
  2. Compară cu hash_equals(expected, received) — previne session hijacking
  3. Caută conversația activă pe canal + session_id
  4. Verifică expirare: dacă ultimul mesaj e mai vechi de 10 minute → marchează conversația ca completed, creează una nouă

Sesiune nouă

UUID v4 generat → HMAC token calculat → Conversation::create() cu tenant_id, bot_id, channel_id, IP, user_agent, origin header.

1.4 Salvare mesaj user

Message::create({conversation_id, direction: 'inbound', content, sent_at}) → increment conversation.messages_count.

1.5 DECIZIA CRITICĂ: Ce tip de query este?

Aici algoritmul decide ce pipeline de search să folosească. Ordinea de verificare:

Pas 1: Este o întrebare despre comandă?

Serviciu: OrderLookupService::detectOrderQuery()

Regex patterns verificate (toate pe mesajul lowercase):

/comand[aă]/   /livr[aă]r/   /colet/   /tracking/   /awb/   /cand.*vine/   /status.*comand/

Dacă DA → apelează OrderLookupService::lookup() care face HTTP request la WooCommerce REST API:

GET {site_url}/wp-json/wc/v3/orders/{order_number}
Authorization: Basic {consumer_key:consumer_secret}

Cache: 10 minute. Rate limit: 30 req/60s per bot. Suportă lookup pe: order number, email, phone.

Context generat pentru AI: [INFORMAȚII COMANDĂ - răspunde pe baza acestor date] Comanda #1234 | Status: processing | Data: 2026-03-20 | Total: 450 RON | Plata: card | Livrare: FanCourier | AWB: 12345678 | Produse: Glet Fino x2, Adeziv CM11 x1

Dacă NU e comandă, dar bot-ul tocmai a cerut detalii:

Verifică dacă ultimul mesaj outbound conține "numărul comenzii" / "emailul" / "telefonul". Dacă da → extrage parametri cu extractOrderParams() (regex pe numere 3-8 cifre, email, telefon).

Pas 2: Este o recomandare de categorie?

Serviciu: IntentDetectionService::detect() → verifică is_category_recommendation

Condiție: NU trebuie să fie deja product_search (are /caut/, /pret/, etc.) ȘI trebuie să match-uie:

/ce\s+(imi|îmi)\s+(recoman|trebui|sugere)/     "ce imi recomanzi"
/ce\s+(produse|materiale)\s+(am|aveti)/          "ce produse aveti"
/ce\s+(am|as)\s+nevoie/                          "ce am nevoie"
/vreau\s+sa\s+(renovez|zugrav|vopsesc|montez)/   "vreau sa renovez"
/cum\s+sa\s+(zugrav|montez|repar|izol)/          "cum sa zugravesc"
/ce\s+materiale/                                 "ce materiale"
/lista.*materiale/                               "lista de materiale"

Dacă DA → extractRecommendationConcept() extrage conceptul (ex: "zugravit", "baie") → RecommendationService::recommend()

RecommendationService — cum funcționează

Menține un map static de ~30 concepte → sub-queries:

"zugravit"  → ["vopsea lavabila", "amorsa", "glet", "trafalet", "banda mascare"]
"baie"      → ["silicon sanitar", "adeziv gresie", "faianta", "gresie", "chit rosturi"]
"termoizolatie" → ["polistiren", "vata minerala", "adeziv polistiren", "plasa fibra sticla"]
"instalatii sanitare" → ["teava ppr", "robinet", "racord", "silicon sanitar"]

Matching: exact match, apoi prefix match pe primele 4 caractere.

Execuție: pentru fiecare sub-query, apelează ProductSearchService::search(botId, subQuery, limit=2). Agregă rezultatele, deduplică pe product ID.

Context generat pentru AI: [Clientul a cerut recomandări pentru "zugravit". Am găsit 10 produse relevante din categoriile: vopsea lavabila, amorsa, glet, trafalet, banda mascare. Produsele se afișează ca carduri. Explică pe scurt DE CE are nevoie de aceste tipuri de produse.]

Pas 3: Product search direct (default)

Dacă nu e nici order, nici recommendation → ProductSearchService::search(botId, userMessage, 4)

Acesta este algoritmul cel mai complex — detaliat complet în Partea III.

Context generat pentru AI (dacă s-au găsit produse): [Am găsit 4 produse relevante care se afișează automat ca carduri sub mesajul tău. NU le enumera în text.]
Context generat (dacă NU s-au găsit): [NU am găsit produse relevante. NU spune că ai găsit produse. Sugerează-i să reformuleze sau să contacteze magazinul.]

1.6 Generarea Răspunsului AI

Metodă: generateAIResponse(bot, conversation, userMessage, extraContext)

Pas 1: Prompt versioning (A/B testing)

BotPromptVersion::selectForBot(botId) — dacă există versiuni active, face weighted random selection pe câmpul weight. Dacă nu → folosește bot.system_prompt.

Pas 2: System prompt (cached 10 minute per bot)

Cache key: bot_system_prompt_{botId}

Conținut:

  1. Base prompt: din BotPromptVersion sau bot.system_prompt sau default "Ești un asistent virtual. Răspunde scurt și util."
  2. Product rules (doar dacă WooCommerceProduct::where('bot_id', botId)->exists()): 
    "Ești asistentul unui magazin online. REGULI STRICTE:
- Produsele se afișează AUTOMAT ca și carduri vizuale sub mesajul tău.
- NU enumera produse în text. NU scrie nume de produse, prețuri, sau liste.
- Răspunde DOAR cu o descriere generală scurtă (1-2 propoziții).
- NU inventa produse, prețuri sau calcule de consum."

Pas 3: Intent detection — skip knowledge?

IntentDetectionService::shouldSkipKnowledge() returnează true dacă mesajul este:

  • Greeting: "salut", "buna", "hello" (max 4 cuvinte, intersecție cu lista)
  • Thanks: conține "multumesc", "mersi", "merci", "thanks"
  • Followup: "da", "nu", "ok", "bine", "perfect" SAU max 3 cuvinte

Dacă skipKnowledge = true → NU se apelează KnowledgeSearchService (economisește un API call OpenAI embedding).

Pas 4: Knowledge Base search (RAG)

Dacă skipKnowledge = false:

KnowledgeSearchService::buildContext(botId, userMessage, searchLimit)

searchLimit = bot.knowledge_search_limit sau 8 (dacă are produse) / 5 (fără produse)

Detaliat complet în Partea IV.

Pas 5: Asamblare prompt final

Ordinea exactă (de sus în jos = de la mai puțin prioritar la mai prioritar pentru LLM):

1. Base system prompt + product rules          (cached 10min)
2. + Knowledge context RAG                      (dinamic per mesaj)
3. + Order context / Product context            (dinamic per mesaj)
4. + PromptGuardrails::apply()                  (ULTIMUL — prioritate maximă)

PromptGuardrails — textul exact injectat

2.1 Inițializare sesiune voice

POST /api/v1/bots/{bot}/realtime-session → RealtimeSessionController::create()

Validări pre-sesiune:

  1. PlanLimitService::canStartVoiceCall($tenant) — verifică minute voice rămase
  2. API key OpenAI validă (nu goală, nu sk-your)

Voice mapping:

Bot-ul are câmpul voice (ex: "feminin", "coral", "ash"). Se mapează la OpenAI voice name:

masculin → ash | feminin → coral | nova → coral | alloy → alloy | echo → echo | shimmer → shimmer

Cloned voice detection:

Verifică $bot->usesClonedVoice() — dacă da, se folosește ElevenLabs TTS în loc de OpenAI native audio.

Construirea instructions (system prompt voice):

  1. Base prompt: bot.system_prompt
  2. Product info (dacă are WooCommerce):

  4. PromptGuardrails::apply(isVoice: true) — adaugă și reguli vocale:

- Răspunde SCURT: maxim 1-2 propoziții simple și directe.
- NU enumera liste lungi. Menționează 2-3 și întreabă dacă vrea mai multe.
- Când spui prețuri, pronunță-le în cuvinte: "douăzeci și cinci de lei", nu "25 RON".
- Dacă mesajul transcris nu are sens, cere politicos reformularea.
  5. Voice-specific rules (hardcodate în controller):

- Răspunde SCURT — maxim 1-2 propoziții
- Ton PROFESIONAL dar cald
- INTERZIS: 'hmm', 'păi', 'deci', 'aș putea', 'cred că'
- Termină cu întrebare scurtă

2.2 Mid-call context update

Serviciu: RealtimeSession::handleConversationEvent()

Trigger: event OpenAI conversation.item.input_audio_transcription.completed (Whisper a transcris vocea userului)

Flow exact:

  1. Salvare transcript: Transcript::create({call_id, role: 'user', content: transcript, timestamp_ms})
  2. Product search (DOAR dacă bot are produse):

PARTEA III — PRODUCT SEARCH (detaliu complet)

Serviciu: ProductSearchService::search(). Zero API-uri externe. Doar PostgreSQL queries.

3.1 Normalizare query

Input: "Ma intereseaza polistiren de 10cm"

  1. Remove diacritics: ă→a, î→i, ș→s, ț→t → "ma intereseaza polistiren de 10cm"
  2. Lowercase
  3. Split compounds: 10cm → 10 cm (regex: /(\d+)(cm|mm|m|kg|ml|l|mp|g)/)
  4. Normalize dimensions: 60 x 60 → 60x60
  5. Collapse whitespace

Output: "ma intereseaza polistiren de 10 cm"

3.2 Token extraction

Tokenizare pe spațiu → stopword removal (100+ cuvinte) → filter: len > 2 SAU conține cifre SAU len=2 alfanumeric

"ma" → stopword ❌ | "intereseaza" → stopword ❌ | "polistiren" → ✅ | "de" → stopword ❌ | "10" → ✅ (cifră) | "cm" → ✅ (2 chars alfanumeric)

Output: ["polistiren", "10", "cm"]

3.3 Token classification

Iterare prin tokens în ordine:

TokenTestClasificare
polistirenNu e numeric, nu e unit, len ≥ 3, productType == nullproduct_type = "polistiren"
10/^\d+$/ match. Next token = "cm" (unit) → perechedimensions = [{value:"10", unit:"cm"}]
cmin_array($units). Prev token = "10" (numeric) → consumedConsumed by dimension
Excepție "CM 11": "cm" e unit, dar next token e "11" (numeric) ȘI prev token NU e numeric → detectat ca product code prefix. Devine product_type = "cm", nu dimension.

3.4 SQL Candidate Retrieval

Un singur query PostgreSQL care caută în 5 câmpuri simultan:

Pentru fiecare token se generează:

  • Name match (2 variante): LOWER(name) LIKE '%polistiren%' + varianta fără diacritice
  • Category match: LOWER(categories::text) LIKE '%polistiren%'
  • Attribute match: LOWER(COALESCE(attributes::text, '')) LIKE '%polistiren%'
  • Description match: LOWER(COALESCE(short_description, '')) LIKE '%polistiren%'

Stemming românesc pe name pattern:

buildNamePattern("suruburi") → stemRomanian("suruburi") = "surub" → pattern = %surub%

Sufixe verificate (în ordine, longest first): urilor, urile, ilor, elor, uri, ele, ile, ari, eri, iri, ati, eti, iti, ul, le, ii, i, e, a

Condiție: stem.length ≥ 3 și word.length > suffix.length + 2

Product type priority în SQL:

Dacă productType setat, se generează un CASE suplimentar: CASE WHEN LOWER(name) LIKE '%polistiren%' THEN 1 ELSE 0 END AS type_in_name

Și se include PRIMUL în ORDER BY: ORDER BY type_in_name DESC, name_match DESC, full_match DESC, trigram DESC

WHERE clause:

WHERE bot_id = :bot_id
  AND stock_status IN ('instock', 'onbackorder')
  AND (
    (LOWER(name) LIKE '%polistiren%' OR ...) -- name match any token
    OR (LOWER(categories::text) LIKE '%polistiren%' OR ...) -- category match
    OR (LOWER(attributes::text) LIKE '%polistiren%' OR ...) -- attribute match
    OR similarity(name, 'polistiren 10 cm') >= 0.3 -- trigram fallback
  )

LIMIT: max(limit * 5, 50)

Se iau mai mulți candidați decât necesari — semantic filter-ul va reduce.

3.5 Semantic Filter (per candidat)

Fiecare candidat trece prin 4 reguli. Score-ul se acumulează.

Rule A — Product Type Gating (OBLIGATORIU)

Dacă productType setat (ex: "adeziv"):

  1. Generează variante: [productType, stemRomanian(productType)] → ex: ["adeziv", "adeziv"] (fără stemming dacă e scurt)
  2. Verifică fiecare variantă în:
    • Name (cu și fără diacritice): str_contains("adeziv cm 11 5kg", "adeziv") → +5 points
    • Identity attributes (Tip produs, Brand, Utilizare, Material, Culoare — 20+ chei): ex: {"Tip produs": ["Adeziv"]} → +4 points
    • Category: str_contains("Siliconi și adezivi", "adeziv") → +3 points
  3. Dacă NU găsit nicăieri → EXCLUDE (nu continuă la alte reguli)
Efect: "Cifră 8 cu Adeziv" conține "adeziv" în name → primește +5. Dar dacă query era "adeziv gresie", Rule C va da bonus doar dacă "gresie" e în atribute. Cifra nu va avea "gresie" → score mai mic → apare mai jos.

Rule B — Dimension Match

Dacă query are dimensiuni (ex: value="10", unit="cm"):

  1. Verifică dacă "10" apare în name (str_contains)
  2. Verifică dacă "10" apare în dimension attributes (grosime, lungime, diametru, etc. — 15+ chei). Dacă unit specificat, verifică și match-ul unitului în atribut sau în key name.
  3. Match → +3 per dimensiune
  4. Niciun match → -3 penalizare (dar NU exclude)

Rule C — Context/Usage Match

Pentru fiecare context word (ex: "gresie", "exterior"):

  • În name → +2
  • În attributes/description → +1
  • În category → +1
  • Absent → 0 (nicio penalizare — e preferință, nu cerință)

Rule D — Usage Compatibility

Extrage valori din atributele: utilizare, aplicare, destinatie, uz

  • Context word găsit în usage values → +2
  • Query cere "exterior" dar produs are doar "interior" → -3
  • Query cere "interior" dar produs are "exterior" → -1 (mild, exteriorul merge și la interior)
  • Fără info usage → 0

Base scoring:

  • Trigram similarity × 2, max 1.0 → contribuție maximă +1
  • Toate tokenurile match-uite (full_words_matched == totalTokens) → +2
  • Partial coverage (>0 dar nu all) → +(matched/total × 1.5)

3.6 Confidence Gate

min_confidence_score = 5 (din config)

Orice candidat cu score < 5 → filtrat. Dacă 0 candidați trec:

  • Dacă ≤ 2 tokens → încearcă spelling correction (Levenshtein ≤ 1 pe primary word, apoi re-search)
  • Dacă > 2 tokens → return [] (0 rezultate)

NICIODATĂ best-sellers random.

3.7 Exemplu complet traced

Query: "adeziv gresie exterior"

Normalizare: "adeziv gresie exterior" (fără schimbări)
Tokens: ["adeziv", "gresie", "exterior"]
product_type: "adeziv" | context: ["gresie", "exterior"] | dimensions: []

SQL: caută unde name/categories/attributes conțin "adeziv" OR "gresie" OR "exterior" OR trigram ≥ 0.3. ORDER BY type_in_name DESC. LIMIT 50.

Candidat: "Adeziv CM 11 5KG"
• Rule A: "adeziv" în name → +5
• Rule B: no dimensions → skip
• Rule C: "gresie" în attributes {"Utilizare": ["gresie"]}? DA → +1 | "exterior" în attrs? NU → 0
• Rule D: usage "gresie" match → +2
• Base: partial_coverage(2/3) = +1.0, trigram(0.21) = +0.42
• Total: 9.42 ✅

Candidat: "Cifră 8 cu Adeziv 45MM"
• Rule A: "adeziv" în name → +5
• Rule C: "gresie" nicăieri → 0 | "exterior" nicăieri → 0
• Rule D: no usage attrs → 0
• Base: partial_coverage(1/3) = +0.5, trigram(0.08) = +0.16
• Total: 5.66 → trece pragul 5 dar va fi JOS în ranking

Candidat: "Parchet Stejar 8MM"

PARTEA IV — KNOWLEDGE BASE RAG (detaliu complet)

Serviciu: KnowledgeSearchService. API extern: OpenAI Embeddings.

4.1 Query expansion (sinonime)

Dacă KNOWLEDGE_QUERY_EXPANSION=true:

Optional LLM rewrite (KNOWLEDGE_LLM_QUERY_REWRITE=true): apel GPT-4o-mini cu prompt "Rescrie query-ul pentru search vectorial, max 15 cuvinte". Cost: ~$0.0005/query.

Dacă LLM rewrite disabled sau fail → dicționar static din config('knowledge.synonyms'):

"pret" → "cost tarif pret"  |  "livrare" → "transport expediere curier livrare"
"garantie" → "garantie retur schimb"  |  "plata" → "platesc achit card numerar"
"contact" → "telefon email adresa locatie"  |  "instalare" → "montaj montare instalare aplicare"

Matching: normalize diacritics pe query word → lookup exact în dicționar → merge synonyms.

4.2 Query embedding

API call extern:

OpenAI::embeddings()->create({
  model: "text-embedding-3-small",
  input: "adeziv gresie exterior cost tarif"  // expanded query
})
→ vector de 1536 dimensiuni

Cache: query_embedding_{botId}_{version}_{md5(query)} — TTL 24h. Dacă API fail → returnează cached dacă există, altfel FTS-only fallback.

4.3 Hybrid search (SQL cu CTE-uri)

CTE 1: vector_search

SELECT id, title, content, chunk_index, source_type,
       1 - (embedding <=> :query_embedding) AS similarity,  -- cosine distance
       ROW_NUMBER() OVER (ORDER BY embedding <=> :query_embedding) AS rank_v
FROM bot_knowledge
WHERE bot_id = :bot_id AND status = 'ready' AND embedding IS NOT NULL
ORDER BY embedding <=> :query_embedding
LIMIT 20  -- candidateLimit

Index folosit: bot_knowledge_embedding_hnsw_idx (HNSW, m=16, ef_construction=64)

CTE 2: fts_search

SELECT id, title, content, chunk_index, source_type,
       ts_rank_cd(to_tsvector('romanian', title), to_tsquery('romanian', :query)) * 3.0
       + ts_rank_cd(to_tsvector('romanian', content), to_tsquery('romanian', :query)) AS fts_rank,
       ROW_NUMBER() OVER (...) AS rank_f
FROM bot_knowledge
WHERE bot_id = :bot_id AND status = 'ready'
  AND to_tsvector('romanian', title || ' ' || content) @@ to_tsquery('romanian', :query)

Config FTS: romanian — stemming nativ PostgreSQL (produsele→prod, livrarea→livr). Title match ponderat 3x.

Index folosit: bot_knowledge_fts_gin_idx (GIN pe tsvector, partial WHERE status='ready')

Merge: Reciprocal Rank Fusion

RRF_score = 1/(60 + rank_vector) + 1.5/(60 + rank_fts)    -- fts_weight = 1.5

FULL OUTER JOIN pe ID → se combină rezultatele din ambele surse.

Post-processing:

  1. Quality filter: similarity ≥ 0.68 OR rrf_score > 1/61
  2. Dedup: max 3 chunks per document (title). Configurabil: KNOWLEDGE_MAX_CHUNKS_PER_DOC=3
  3. Optional LLM reranking (KNOWLEDGE_RERANKING=true): trimite top candidați la GPT-4o-mini cu prompt "Return indices of most relevant passages". Cost: ~$0.002/query
  4. Content dedup: elimină chunks cu >80% word overlap (array_intersect pe primele 100 cuvinte unice)
  5. Group & sort: grupează pe document, sortează intern pe chunk_index
  6. Budget: concatenează chunks până la max 6000 chars (configurabil)

Format output context:

Informații relevante din baza de cunoștințe:

--- Politica de livrare [connector] (relevance: 85%) ---
Livrarea se face în 24-48 ore lucrătoare prin curier...

--- Informații contact [url] (relevance: 72%) ---
Program: Luni-Vineri 08:00-17:00...
