Final Recommendation: Option C (Hybrid)

Architecture:
  • Vector search layer: SHARED. OpenAI embeddings are multilingual by design. text-embedding-3-small/large handles Romanian and English equally well at the vector level. One HNSW index, one embedding model, one search query. No duplication needed.
  • FTS layer: LANGUAGE-SPECIFIC. PostgreSQL FTS configs differ by language (stemming, stopwords). Store a language column on bot_knowledge. Use to_tsvector(language, text) dynamically. Create partial GIN indexes per language for tenants with significant volume.
  • Query expansion: LANGUAGE-SPECIFIC. Synonym dictionaries per language. Language detected from bot.language setting (not runtime detection for known bots).
  • Caching: LANGUAGE-AWARE keys. Include language in cache keys to prevent cross-language cache pollution.

Specific multilingual concerns

ConcernSolution
Romanian FTS vs English FTSUse bot's language field to select FTS config. to_tsvector($lang, text). PostgreSQL has built-in romanian and english configs with correct stemming.
Synonyms dictionary per languageStructure config as knowledge.synonyms.ro and knowledge.synonyms.en. Load based on bot language.
Query expansion per languageLLM rewrite prompt includes language instruction. Dictionary lookup uses language-specific map.
Mixed-language contentDetect at chunk level during ingestion. Store content_language on each chunk. Vector search handles mixed content naturally; FTS uses detected language or falls back to simple config.
Product titles in different languagesWooCommerce content comes in the store's language. Store as-is. Vector embeddings handle cross-language similarity. FTS only matches same-language queries.
Bot UI language differs from content languageVector search bridges this gap (semantic similarity is cross-lingual). FTS may miss, but that's acceptable since vector search is the primary retrieval method. RRF scoring naturally handles the imbalance.

P4 Products as First-Class Retrieval Objects

WooCommerce products are NOT just catalog records. They are semantic knowledge that must support exact lookup, recommendations, comparisons, suitability questions, and attribute filtering.

Product Field Strategy

FieldEmbedded (semantic)?Structured (filter)?Notes
Name / TitleYesYesPrimary identifier. Weighted 3x in FTS.
Short descriptionYesNoHigh-value semantic content.
Full descriptionYes (truncated 1500 chars)NoTruncate to avoid noise. Strip HTML.
CategoriesYes (as text)YesInclude full path: "Electronics > Laptops > Gaming"
Attributes (brand, color, material, size)Yes (as "key: value" pairs)YesNormalize: "Brand: Samsung, Culoare: Negru"
PriceNoYesNEVER embed price. Changes too frequently. Inject at query time from live data.
Sale priceNoYesSame. Live data only.
Stock statusNoYesNEVER embed stock. Check at query time. Optionally filter out OOS from results.
SKUNoYesExact match only.
ImagesNoYes (URLs)For display in chat cards.
PermalinkNoYesFor "View product" links.

Embedding Template for Products

Produs: {name} Categorie: {category_path} {short_description} {description_truncated_1500} {foreach attribute: "{attr_name}: {attr_values}"} // DO NOT include: price, stock, SKU, URLs // These are injected from structured data at query time

Product Retrieval Modes

ModeMethodExample query
Exact lookupStructured: SKU match, name match (trigram)"Arată-mi produsul X" / "SKU ABC123"
Category browsingStructured: filter by category_id"Ce laptopuri aveți?"
Attribute filteringStructured: WHERE attribute = value"Tricouri roșii mărimea L"
Semantic recommendationVector search on product embedding"Ceva pentru cadou de Crăciun"
ComparisonVector search + structured data side-by-side"Diferența între X și Y"
SuitabilityVector search + attribute filtering"Ce parfum e bun pentru birou?"
Critical design rule: Price and stock must NEVER be part of the embedding. They change frequently and would require constant re-embedding. Instead, inject live price/stock data into the LLM context AFTER retrieval, from the structured woocommerce_products table. The embedding captures semantic identity; the structured data provides fresh transactional facts.

Avoiding Noisy / Duplicate Product Content

  • HTML stripping: Already implemented. Good.
  • Description truncation: 1500 chars max. Prevents SEO-stuffed descriptions from polluting embeddings.
  • Duplicate detection: Use content_hash (SHA-256) at ingest. If product content hasn't changed, skip re-embedding.
  • Category dedup: A product in 3 categories should NOT create 3 knowledge entries. One entry with all categories concatenated.
  • Variant handling: WooCommerce variable products should embed the parent only. Variations stored as structured attributes. Prevents 50 near-identical embeddings for color variants.

P9 Final Recommendation

1. Cost Impact Summary

  • RAG search costs are negligible (~1.4% of total message cost). Optimizing RAG infrastructure has near-zero cost impact.
  • The biggest cost lever is chat completion routing. If better retrieval quality lets 5% more queries use gpt-4o-mini instead of Claude, you save $0.60/1K messages (21% reduction).
  • All recommended improvements combined add <$0.10/1K in infrastructure cost while potentially saving $0.60/1K through better routing.
  • Net expected saving: ~$0.50/1K messages ($2.89 → ~$2.39).

2. Final Multilingual Strategy

Option C: Hybrid approach.
  • Shared vector layer (embeddings are multilingual by design).
  • Language-specific FTS configs (romanian, english, simple fallback).
  • Language-specific synonym dictionaries.
  • Language stored per chunk (content_language column).
  • Bot language used as primary signal, not runtime detection.
  • Cost: $0 extra. Effort: ~8 hours.

3. Final Retrieval Architecture

4-layer retrieval with intelligent routing:
  • Layer 1: Structured product retrieval (SQL exact/filter) for specific product queries.
  • Layer 2: Semantic product retrieval (vector search on product embeddings) for recommendations/suitability.
  • Layer 3: General knowledge RAG (hybrid vector+FTS) for FAQ, policies, documentation.
  • Layer 4: Order lookup (WooCommerce API) for order status queries.
  • Router: IntentOrchestratorService decides which layers to activate per message.
  • Assembler: ContextBudgetService merges, deduplicates, and truncates to channel budget.

7. Rollout / Migration Plan

  • Phase 1 (Quick wins): Deploy immediately. All changes are additive (new columns, new config). No breaking changes. Existing data works as before.
  • Phase 2 (Multilingual): Backfill content_language column with 'ro' for all existing data. New ingestion sets it automatically. No re-embedding needed.
  • Phase 3 (Product retrieval): Generate product embeddings as background job. Doesn't affect existing product search. New semantic retrieval runs in parallel. Gradual rollout per tenant.
  • Phase 4 (Embedding upgrade): Create new embedding_v2 vector(3072) column. Re-embed in background. Switch search to v2 when complete. Drop old column after verification. Zero-downtime migration.

Do the 5 quick wins this week. They cost $0, reduce latency, and improve quality.
