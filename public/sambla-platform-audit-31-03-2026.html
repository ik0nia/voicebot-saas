What Is Already Strong

Hybrid RAG Pipeline: The vector + FTS + RRF fusion, business-aware rescoring, dedup, overlap-aware chunking, LLM reranking, and Romanian-specific FTS is genuinely above market for this segment. Most competitors use naive vector-only search.
Multi-Intent Orchestration: The transition from legacy single-intent to scored multi-intent with pipelines is well-architected. Priority-based execution, contextual follow-up detection, and data-source gating are production-grade.
Voice + Chat Parity: Same AI brain across both channels with appropriate per-channel prompt tuning (voice brevity vs chat structure). Cloned voice support adds premium differentiation.
Lead Pipeline & CRM: Full pipeline (new → contacted → scheduled → met → quoted → won/lost) with callback scheduling, opportunity scoring, and conversation-to-lead attribution. This is a complete CRM mini-system built into the platform.
E-commerce Integration Depth: Product search with semantic filtering, dimension matching, order lookup via WooCommerce API, add-to-cart tracking, purchase attribution, and funnel analytics. Goes well beyond typical chatbot integrations.

What Is Limiting Quality

ConversationPolicy Not Connected to Prompt Builder: The ConversationPolicy model exists with rich fields (tone, verbosity, CTA aggressiveness, brand vocabulary, custom messages, business rules) but is never read by PromptBuilder. This is the single biggest quality gap — all the tenant customization infrastructure exists but isn't wired up.
Chat Channel Missing Orchestrator: The advanced multi-intent orchestrator (IntentOrchestratorService) is only used by voice. Chat messages go through ChannelMessageService which uses the legacy IntentDetectionService — no confidence scoring, no data-source gating, no complexity classification. Chat quality is systematically lower than voice.
No Guided Onboarding: A tenant must figure out on their own what to put in the knowledge base, how to write a good system prompt, and what settings matter. There's no wizard, no templates, no quality scoring. The path from "signed up" to "good results" is unclear.

What Is Missing to Sell

No Bot Health Score: Tenants can't see if their bot is well-configured. A simple "Bot Health: 72%" with "Add FAQ content" suggestions would drive self-service quality.
No Demo Mode / Playground: No easy way for a buyer to test a bot with pre-loaded demo content before committing. The existing /demo/{slug} page is voice-only and requires a configured bot.
No Preset Templates: No "E-commerce Bot" or "Service Business Bot" template that pre-fills system prompt, greeting, KB structure, and settings. Every tenant starts from scratch.

2. End-to-End AI Flow Audit

Stage 1: User Input

How It Works

Chat: Widget sends message via POST /api/v1/chatbot/{channel}/message. Voice: OpenAI Realtime API receives audio, Whisper transcribes, triggers conversation.item.input_audio_transcription.completed.

Good: Rate limiting (30/min chat, per-IP), session management with HMAC tokens, domain verification for widget CORS.

Weak: Voice transcription depends entirely on Whisper quality. The WhisperHallucinationFilter catches 40+ patterns but is static — no feedback loop from corrections.

Recommendation: Log filtered hallucinations per bot to identify patterns specific to a tenant's domain. Allow tenants to add custom "ignore phrases" (e.g., background music patterns in their call center).

Stage 2: Intent Detection

How It Works

Voice (Orchestrator): Multi-intent with confidence scoring (0.0–1.0), 8 intent types, contextual follow-up awareness, priority-ordered pipelines.

Chat (Legacy): Boolean flag detection — single pass, no confidence, no context awareness for follow-ups.

Weak: Chat channel is significantly weaker. A user saying "da" after the bot asked "Vrei să comandați?" is not recognized as order confirmation in chat. Regex-only detection (no ML/embeddings) means it's brittle for Romanian variations — "as dori" (I would like) won't trigger isNewOrderIntent.

Critical Gap: Greeting detection is max 4 words hardcoded. "Bună ziua, am o întrebare" (Hello, I have a question) = 6 words → not greeting → hits full RAG pipeline for a simple hello.

Recommendation: Route chat through the Orchestrator. Increase greeting word limit to 8. Add "as dori", "aș vrea", "m-ar interesa" to new_order patterns.

Stage 3: Query Rewrite / Classification

How It Works

Query expansion via Romanian synonym dictionary (12 groups) + optional LLM rewrite (gpt-4o-mini, now enabled). Complexity classification: simple/medium/complex based on word count + signal words.

Good: LLM rewrite transforms vague queries ("cât costă ăla") into search-optimized text. Dictionary expansion covers common Romanian e-commerce terms.

Weak: Dictionary only has 12 synonym groups — missing common queries like "disponibilitate" (availability), "termen" (deadline), "material" (material type), "compatibil" (compatible). LLM rewrite adds ~200ms latency on every query even when unnecessary (simple queries).

Recommendation: Skip LLM rewrite for simple complexity queries. Expand synonym dictionary to 25+ groups covering construction/home improvement domain.

Stage 4: Retrieval

How It Works

Hybrid search: pgvector HNSW cosine similarity + PostgreSQL GIN FTS (Romanian stemmer) merged via RRF scoring. Up to 20 candidates retrieved when reranking enabled.

Good: HNSW index (m=16, ef=64) provides excellent recall. Romanian-specific FTS catches keyword matches that vector search misses. RRF fusion is well-implemented with configurable FTS weight (1.5x).

Weak: Similarity threshold at 0.55 may include noisy results that burden the reranker. No query-type adaptive retrieval — a product question and a policy question use the same pipeline with the same limits.

Stage 5: Reranking

How It Works

LLM reranking (gpt-4o-mini, now enabled) on deduplicated candidates. Business-aware prompt prioritizes direct answers over tangential context.

Good: 350-char snippets give reranker enough context. Business-aware prompt is a significant improvement over generic "relevance judge".

Weak: Reranker sees title + 350 chars only — may miss relevant content in the middle/end of longer chunks. No cross-encoder or dedicated reranking model (gpt-4o-mini is general-purpose). Cost: ~$0.002/query — acceptable but adds up at scale.

Stage 6: Prompt Building

How It Works

PromptBuilder assembles: base prompt + products + knowledge + summary + extra context + order rules + response quality + confidence modifier + fallback + guardrails. Token budget enforced via ContextBudgetService.

Good: Clear priority hierarchy (products > knowledge > summary > history). Guardrails always appended last (highest LLM priority). Voice vs chat prompt differentiation.

Critical Gap: ConversationPolicy (tone, verbosity, CTA aggressiveness, brand vocabulary, custom greeting/handoff/lead/out-of-stock messages, prohibited phrases, business rules, snippets) is never loaded by PromptBuilder. All that rich configuration exists in the database but has zero effect on AI behavior.

Recommendation: Wire ConversationPolicy into PromptBuilder.build(). This single change would unlock massive configurability for tenants.

Stage 7: Answer Generation

How It Works

Chat: ChatCompletionService via OpenAI API (model routed by ChatModelRouter). Voice: OpenAI Realtime API (GPT-4o) with streaming audio output.

Good: Model routing allows cost optimization. Conversation summarization kicks in after 8 messages to prevent context overflow.

Weak: No streaming for chat (user waits for full response). Summarization prompt is basic ("max 150 words") — may lose important details like specific products discussed or prices mentioned.

Stage 8: Follow-up Behavior

How It Works

Voice orchestrator checks last bot message for contextual cues ("doriți să comandați?"). Chat: no follow-up awareness — each message treated in isolation (except conversation history).

Weak: If the bot asks "Vrei în alt culori?" and user says "da" in chat, the system has no memory of what "da" refers to. Voice handles this via orchestrator context, chat doesn't.

Stage 9: Lead Capture / Handoff

How It Works

Lead scoring based on engagement signals (message count, product clicks, intent types). Threshold: 30 points. Voice: automatic phone/name extraction from transcripts. Chat: regex extraction after 3+ messages. Handoff: detected by keyword patterns ("operator", "agent", "persoana").

Good: Sophisticated scoring with positive and negative signals. Voice lead extraction handles spaced digits ("0 7 4 2...") and name confirmations.

Weak: Lead capture timing is hardcoded — can't be adjusted per tenant. Some businesses want early aggressive capture, others want consultative approach first. The lead_aggressiveness field in ConversationPolicy exists but isn't used.

3. RAG Audit

Indexing Quality

Embedding Model: text-embedding-3-small (1536 dim) — good balance of cost and quality for Romanian. HNSW index with m=16, ef_construction=64 provides ~98% recall.
No Embedding Model Upgrade Path: When OpenAI releases better embedding models, there's no mechanism to re-embed existing chunks. The embedding_model column tracks version per chunk but no migration tooling exists.

Chunking Strategy

Source TypeChunk Size (tokens)OverlapQuality Filter
FAQ12816 tokens (12.5%)Min 20 chars / 3 words
Manual25632 tokensMin 50 chars / 5 words
Scan38448 tokensMin 50 chars / 5 words
Upload / Connector / Agent51264 tokensMin 50 chars / 5 words
Good: Semantic paragraph-boundary splitting (not arbitrary character cuts). Source-type-aware chunk sizes. Token-accurate counting via tiktoken. Overlap prevents information loss at boundaries.
Issue: No semantic chunking — splitting is purely by paragraph + token count. A topic that spans 3 paragraphs gets split into separate chunks that lose coherence. For FAQ content at 128 tokens, multi-part answers get fragmented.

Metadata Usage

Underused: Chunks have source_type, category, source_date, metadata (JSON) fields, but only source_type is used in business-aware rescoring. Category filtering exists in the search API but tenants have no UI to assign categories. The metadata JSON field is unused.

Context Injection

Well-designed: Chunks grouped by source title for coherent reading, sorted by chunk_index within groups. Near-duplicate removal (>80% word overlap). Budget-aware truncation.
Missing source attribution in context: The context block shows titles and relevance percentage but doesn't indicate if information is from FAQ (authoritative) vs. website scan (potentially outdated). Tenants can't prioritize source types.

Failure Modes & Hallucination Risks

Quality Gate Too Binary: If top similarity < 0.50 AND top RRF < 0.025, the system returns ZERO context. This means the bot either has good context or none at all — no "partial confidence" mode where it says "I found some related information but I'm not fully sure."
Stale Content Risk: Website scan content has no expiration. A bot scanning a website once gets that content forever, even if the website changes. No re-scan scheduling or staleness detection.

4. Prompt System Audit

Architecture Assessment

Guardrails Last Pattern: PromptGuardrails::apply() appends anti-hallucination rules at the END of the prompt — maximizing LLM attention. This is correct prompt engineering.
Confidence-Adaptive: Low-confidence prompts instruct the bot to be cautious and ask clarifying questions. High-confidence prompts allow definitive answers. This prevents hallucination scaling with certainty.

Prompt Weaknesses

System Prompt Is Single Textarea: The tenant writes ONE system prompt that controls everything — personality, tone, domain knowledge, rules. No separation of concerns. A tenant who writes a good personality prompt but forgets business rules gets a charismatic but dangerous bot.
No Prompt Validation: A tenant can write "Ignore all previous instructions" in the system prompt and there's no sanitization. The guardrails are appended AFTER, which helps, but the base prompt could still contain conflicting instructions.
Hardcoded Romanian: All prompt templates (response quality, fallback, guardrails, order rules, lead capture) are in Romanian. No i18n layer. A tenant setting language: en gets an English-speaking bot with Romanian instruction templates — the bot will respond in English but its internal rules are in Romanian, causing inconsistent behavior.

Proposed Layered Prompt Architecture

Layer Stack (bottom to top = lowest to highest priority)

  1. Global Base Layer: Platform-wide rules (safety, anti-hallucination, data handling). Never overridden by tenant.
  2. Business Type Layer: E-commerce rules (product comparison, pricing), Service rules (booking, consultation), or Hybrid rules. Auto-selected from preset.
  3. Tenant Brand Layer: From ConversationPolicy — tone, verbosity, brand vocabulary, prohibited/required phrases. Written by tenant or generated from template.
  4. Channel Layer: Voice (brevity, no lists, questions), Chat (structured, markdown, cards), WhatsApp (emoji-friendly, shorter).
  5. Task Layer: Dynamic per-request — search prompt, order lookup prompt, lead capture prompt, complaint handling prompt. Injected by orchestrator based on detected intents.
  6. Guardrails Layer: Anti-hallucination, data verification, escalation triggers. Always last. Never overridable.

5. Conversation Logic Audit

Greetings

Voice: Time-of-day-aware greetings ("Bună dimineața" / "Bună ziua" / "Bună seara"). Natural and warm.
Chat: Uses greeting_message from bot config — static, same greeting every time. No time awareness, no returning-visitor detection, no personalization. A user coming back the next day gets the same "Bună! Cu ce te pot ajuta?" without acknowledgment.

Follow-up & Clarification

No Explicit Clarification Logic: When the bot isn't sure what the user means, it relies entirely on the LLM to decide whether to ask a clarifying question. There's no structured "if ambiguous, ask X" logic. The prompt says "pune o întrebare de clarificare" but doesn't define what counts as ambiguous.

Complaint Handling

Detected But Not Specially Routed: IntentDetectionService detects complaints (nemulțumit, reclamație, defect) but the orchestrator doesn't have a dedicated complaint pipeline. Complaints go through the generic knowledge search, which may return product info instead of escalation instructions.

Recommendation: Add a complaint pipeline that: (1) acknowledges the issue empathetically, (2) asks for details, (3) always offers human escalation, (4) never tries to sell during a complaint.

Multi-Intent Handling

Voice orchestrator handles multiple intents well: "Vreau să comand adezivul ăla, dar mai întâi verifică comanda anterioară" → both new_order and existing_order pipelines fire.
Chat can't handle this: Same message in chat gets only one intent detected (first match wins). The user gets a confusing response that addresses only one intent.

Memory / Context Carryover

Good: last_product_context stored in conversation metadata enables "pe ăla vreau" (I want that one) references. Conversation summary compresses history after 8 messages.
Missing: No structured memory of what the user has already been told. If a user asks about shipping twice, they get the same answer repeated verbatim. No "as I mentioned earlier" awareness.

Where It Feels Robotic

  • Identical greetings: Every chat session starts the same way
  • No empathy escalation: After 3 failed searches, no "I'm sorry I haven't been helpful — would you like to speak with someone?"
  • No personality variance: Bot always responds in the same style regardless of conversation mood
  • Static call-to-action: Always pushes to the same action (order/contact) regardless of where the user is in their journey

Recommended Conversation Improvements

  1. Frustration Detection: Track failed searches, repeated questions, complaint words. After threshold → auto-escalate tone (more empathetic) and offer handoff.
  2. Journey-Aware CTAs: First visit → explore. Second visit → compare. Third visit → buy. Adapt CTA based on conversation count for same visitor.
  3. Repeat Question Detection: If user asks the same topic twice, acknowledge it: "Am înțeles că subiectul livrării este important pentru tine. Hai să detaliez mai mult."
  4. Graceful Degradation: When confidence is low, instead of generic fallback, offer specific alternatives: "Nu am găsit exact asta, dar pot să te ajut cu [related topic A] sau [related topic B]?"

7. Knowledge Base Builder Audit

Current State

Multiple Input Methods: File upload (PDF, DOCX, TXT, CSV), URL scraping, raw text input, AI agents (FAQ generator, product sync), website scanner, WooCommerce connector. This is comprehensive.
No Structure or Guidance: The KB builder is a flat list of documents. No categories, no templates, no "you're missing X" warnings. A tenant uploading 10 PDFs has no idea if the content is sufficient for good bot performance.
No Content Quality Validation: Any text is accepted — even a document saying "Lorem ipsum" gets chunked and embedded. No readability check, no duplicate detection across documents, no thin-content warning.

Show a visual score: "Your bot knows 65% of what it needs" with specific gaps highlighted.

Content Quality Rules (Proposed)

  • Warning: Document with <100 words — "This content is very short and may not provide enough context for the AI"
  • Warning: No question-answer pairs detected — "Consider adding FAQ-style content for better bot responses"
  • Warning: No pricing information detected — "Customers frequently ask about prices. Consider adding pricing data."

9. WOW Factor Audit

Top 10 Missing Product Ideas

  1. AI Setup Assistant: "Describe your business in 2 sentences" → AI generates system prompt, greeting, and KB structure suggestions. Reduces setup from hours to minutes.
  2. Live Conversation Takeover: Human agent sees live conversation, can take over mid-chat. Bot summarizes context for the agent automatically.
  3. "What Your Bot Doesn't Know" Dashboard: Show top 20 unanswered questions per week. One-click "Generate answer" using AI agent. This turns a gap into a feature.
  4. Conversation Replay with AI Commentary: Replay a conversation with annotations: "Here the bot used RAG chunk #3" / "Here the bot had low confidence" / "This is where the customer became interested."
  5. Smart Notifications: "Your bot had 5 conversations today where it couldn't find shipping information. Add your shipping policy to fix this." Push via email/Slack.
  6. Voice Cloning 2.0: Clone the business owner's voice from a 30-second sample. "Your customers hear YOU, 24/7." Powerful emotional pitch.
  7. Multi-Bot Routing: One phone number, multiple bots. Caller says "I want to talk about product X" → routed to the right specialized bot.
  8. Scheduled Outbound Campaigns: Bot calls leads from the pipeline automatically. "Call all 'contacted' leads and ask if they're ready to schedule."
  9. WhatsApp Catalog Sync: Bot's product knowledge auto-published as WhatsApp Business catalog. Customer browses + asks questions in same chat.
  10. Customer Satisfaction Auto-Survey: After each conversation, bot asks "A fost util acest răspuns? 1-5" → feeds into quality scoring.

Summary

Sambla is a technically impressive platform with genuine depth in RAG, multi-intent orchestration, and e-commerce integration. The core AI pipeline is above market average. The primary gaps are in product polish: disconnected configuration (ConversationPolicy), chat/voice parity, onboarding guidance, and tenant self-service tools. Closing these gaps transforms a strong engineering project into a sellable product.
